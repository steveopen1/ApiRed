# 黑盒视角下基于AI技术的API安全实践


## 0x01 背景介绍
近年来，攻击面管理(ASM)已成为企业构建安全体系的核心策略之一。ASM工具能够帮助企业从攻击者的视角识别和管理互联网资产，但作为企业互联互通核心纽带的API接口，却往往未得到传统攻击面管理的足够重视和系统梳理。API接口作为企业数据传输和业务交互的关键通道，其安全性直接关系到企业资产和数据的安全性。


## 0x02 为何要做
从攻击者到防守者的身份转变，让我对安全问题有了新的视角。在查看白帽子提交的漏洞报告时让我对API安全有了新的认知，看似简单的漏洞，却可能导致企业核心数据被彻底泄露。


实际工作中遇到的几起典型案例，更让这种认知变得尤为深刻：
- 某系统曾被发现存在大量漏洞，可直接/间接泄露公司全渠道核心信息；
- 某系统存在影子API，导致全公司员工姓名、手机号等敏感数据在未授权情况下可被读取；
- 某平台的多个独立服务，仅通过拼接“/api/admin/actuator”与“/api/user/actuator”路径，就被白帽子成功挖掘出多个Spring Actuator接口泄露隐患；


这些案例让我意识到，若此类数据落入黑产或竞争对手手中，后果将不堪设想。深入分析IT部SRC的全部漏洞后发现，无论是第三方采购系统还是自研系统，攻击根源几乎都指向API安全问题。


进一步与挖掘这些漏洞的白帽子交流后，我总结出他们的核心思路：通过半自动化工具爬取JS、提取API，再结合Burp爆破模块进行组合测试，最后人工看结果一个个筛选。这种方法虽有效，但高度依赖人工临时操作，当面对上百/上千个系统时，显然无法实现常态化防护。


因此，我希望在黑盒视角下引入AI技术，构建主动安全防护策略，通过系统化手段自动发现和梳理企业API接口。通过这种方式，提升整体API的安全性，确保企业资产与数据的安全，全面防范潜在的安全威胁。


## 0x03 效果预估
1. 精准识别API未授权访问：传统方法难以区分API未授权访问的真伪，导致误报率高。可通过AI技术进行检测，降低误报，提升评估可靠性。
2. 覆盖微服务架构安全问题：传统检测难以扫描微服务架构中特定接口的安全问题(如/api/user或/api/admin下的Spring Actuator接口)。通过专门编写的代码逻辑，可以有效覆盖这些接口，避免关键漏洞遗漏。
3. 检测敏感信息泄漏：HTML/JS/API的响应中可能泄露敏感信息(如OSS存储桶密钥泄露、小程序AppSecert泄露等)。通过AI技术或定制化的检测逻辑，可以智能分析响应内容，及时发现并预警敏感信息泄漏，降低数据泄露风险。


## 0x04 难点分析
1. 适配主流前端技术并提取JS资源文件：现代前端框架(如React、Vue、Angular)广泛采用动态加载技术，传统黑盒API安全检测方案难以有效提取这些框架中的JS资源文件。这可能导致数据源不完整，进而影响API接口的全面发现。因此，需要开发能够高度适配这些主流前端技术的解决方案，确保所有JS资源文件都能被完整提取，为后续API接口发现提供可靠的数据基础。
2. 全面发现动态API接口：部分API接口是通过变量拼接或其他动态方式生成的，传统检测方案往往无法识别这类接口，导致安全漏洞漏检。例如，某些API路径可能需要通过前端代码中的变量拼接才能完整生成。这就需要一种能够解析前端代码逻辑、动态追踪API生成过程的检测方法，以确保所有动态API接口都能被充分识别，避免安全盲区。
3. 清晰认知微服务架构：在微服务架构中，一个站点通常包含多个独立服务(如admin、user等)，每个服务都有专属的API接口。由于服务独立部署、技术栈可能存在差异，不同服务的安全状态也可能截然不同，比如A服务不存在第三方组件漏洞，B服务却可能受此类漏洞影响。传统检测方案压根就不识别和区分这些服务前缀，导致对整体服务架构的认知不清晰。因此，需要开发能够自动识别和提取这些服务前缀的检测机制，以全面掌握站点的微服务架构，确保安全检测的完整性。
4. 提高API未授权访问检测的准确性：传统检测方法在识别API接口未授权访问时，容易出现误报或漏报的情况，严重影响安全评估的可靠性。未授权访问检测需要在海量的API调用中精准识别出哪些是真正的安全威胁，哪些是正常的业务行为。这就需要通过结合上下文信息分析以及AI技术来提高检测的准确性和可靠性，避免误判和漏判。
5. 快速准确匹配响应中的敏感信息：HTML/JS/API的响应中可能包含敏感信息(如OSS存储桶密钥泄露、小程序AppSecert泄露等)，传统检测方案往往无法快速、有效地识别和处理这些信息。这就需要开发能够快速扫描并匹配响应内容的检测机制，通过正则表达式、关键词库、AI技术来精准识别敏感信息，及时发现潜在的安全风险，避免数据泄露。


## 0x05 AI的定性
通过前面的分析，我们已经对需要解决的各种难点有了初步的了解。如果仅依靠传统代码来解决问题，会面临诸多困难。那么，如果引入AI技术呢？


每个人对AI的理解和定位可能各不相同，而我所认为的AI的核心价值在于它能够解决传统代码难以触及的复杂场景。AI通过模拟人类思维的灵活性和创造性，为那些需要深度理解与判断的问题提供有效的解决方案。


具体到我这个项目中，用通俗的话来说就是：设想一个工作流，在每个流程节点中明确所需的数据。AI模拟人的身份，从人的视角提取各个步骤中的数据。同时，传统代码作为基础工具，用于兜底处理。最终，我们将这些数据整合并过滤，来完成既定目标。


## 0x06 解决思路
### 0x06.1 全流程工作流设计
以“黑盒视角下的自动化API安全检测”为目标，构建“资源提取→API发现→安全检测→智能分析”的全流程工作流，其中AI负责处理传统代码难以应对的复杂场景(如动态逻辑解析)，传统代码负责标准化、规模化的基础操作(如资源爬取、规则校验)，两者协同实现API安全检测功能。

#### 0x06.3.2 解决方案
经过前面几个示例，我们已经对开发人员构建API的常见结构有了一定认知。实际场景往往复杂多样，像动态拼接路径这类情况，纯代码方案很难完美覆盖，而引入AI技术，就能有效突破这类传统方法的瓶颈，让复杂场景的API识别与分析更顺畅。


基于前面的示例，我意识到有必要对API结构进行规范化拆解。而规范化的核心在于分层梳理，我可以将一个完整的API拆解为基础URL、基础API路径、API路径三个层级，各层级关联逻辑如下：
- 完整API构成：基础URL + 基础API路径 + API路径
- 例：https://xxx.com(基础URL) + /api(基础API路径) + /user/getInfo(API路径) → https://xxx.com/api/user/getInfo(完整路径)


API分层后，对应层级的获取方案也更清晰，让我们能根据不同层级的特点，灵活选用传统代码或AI技术，精准高效地完成各部分的提取工作。


##### 基础URL的获取方法(均用传统代码实现)
1. 从历史HTTP请求记录中提取基础域名URL，去重后直接使用；
2. 从用户输入的URL中提取基础域名，生成该域名下所有可能的路径层级作为基础URL列表；
3. 利用正则表达式提取基础URL。


##### 基础API路径的获取方法
- 传统代码实现：
  1. 根据预设关键字，从网络请求和API路径中提取包含这些关键字的路径作为基础API路径；
  2. 对比首页网络请求路径和已知API路径，从末尾匹配的部分提取基础API路径前缀；
  3. 提取子域名：例如输入api-v2.xxx.com，可生成/api、/v2、/api-v2为基础API路径。
- AI技术辅助：
  提前定义基础API路径的规则，并告知AI完整API的构成逻辑(基础URL + 基础API路径 + API路径)，让AI理解需求后，读取JS并按指定格式输出是否存在基础API路径及具体结果。

### 0x06.4 针对“清晰认知微服务架构”
#### 0x06.4.1 示例
该图呈现了某站点包含的多个独立服务(如/liz-basis-authuser、/liz-qywechat-agent)，示例接口路径：
```javascript
"/liz-qywechat-agent/authsec/component/signature"
```


#### 0x06.4.2 解决方案
基于前期对API结构的规范化拆解，虽覆盖了基础路径解析，但不同服务的差异化安全扫描需求仍难落地。因此进一步细化分层，新增服务路径维度，通过“精准定位服务归属”补足场景能力，关联逻辑如下：
- 服务路径定位：API路径的首个分段
- 例1(结构定位)：/user/getInfo(API路径)中，/user是首个分段，即服务路径
- 例2(功能组合)：https://xxx.com(基础URL) + /api(基础API路径) + /user(服务路径) → 服务级标识(https://xxx.com/api/user)，支撑后续针对不同服务的自定义扫描


**服务路径分层与提取(传统代码实现)**：
从现有的API路径中提取第一个路径分段直接认定为服务路径。


由于该场景的解析规则固定且明确，采用传统代码即可高效实现，因此无需引入AI方案。


### 0x06.5 针对“提高API未授权访问检测的准确性”
#### 0x06.5.1 概述
采用“传统过滤 + AI研判”的多层检测流程，先通过传统代码过滤404相关接口；再由AI依次完成站点定性、接口登录需求研判、结合站点定性判断接口是否为公共接口；最终人工排除公共接口后，剩余接口即为未授权接口。


#### 0x06.5.2 解决方案
##### 提高API未授权访问检测准确性(传统代码实现)
实现第一层过滤：通过代码构造404特征路径(基于基础API路径、服务路径、无效片段组合)，以此作为参照，将所有API请求与其响应特征(如状态码、内容)进行比对，直接排除响应与404一致的相关接口，减少后续处理的无效数据量。


##### 提高API未授权访问检测准确性(AI技术实现)
基于prompt引导AI处理过滤后的接口，分三步完成研判：
1. 站点定性：向AI输入站点首页内容、页面标题、API接口等基础信息，通过prompt指令引导AI综合分析，最终输出对站点业务的描述(例如“该站点为企业内部会议室预约管理系统”)；
2. 接口登录分析：向AI输入API接口的请求路径、响应状态及响应体，通过prompt指令引导其分析，由AI判断该接口是否需要登录才能访问；
3. 公共接口分析：提取经AI研判为无需登录的接口，结合此前的站点定性结果，通过prompt指令引导AI综合分析，判断该接口是否为公共接口。


最终排除公共接口后，剩余的即为未授权访问接口。

示例prompt交互：

> 1. 账号密码类：各类平台的账号(如用户名、邮箱账号、手机号登录账号等)，对应账号的密码(包括明文密码、加密后但可识别为密码的字符串)；

> 3. 应用密钥类：小程序相关密钥(如AppSecret、AppID等，需区分不同平台的小程序密钥)，其他应用的专属密钥(如第三方应用的授权密钥、接口密钥等)。
> 请你按我下面的这个格式来回答我，如果有多个结果在<结果>标签中使用$$$$进行分隔：<思考>分析过程，严格控制在150个字以内</思考>+<判断>有/无/无法确定<判断>+<结果>如果判断为有就直接输出敏感信息在这里，其他情况这里留空</结果>

##### 补充：大体积响应数据的处理方案
在实际开展敏感信息响应匹配工作时，常会遇到“请求响应数据过大”的问题，比如部分响应数据大小可能达到1000KB、2000KB甚至更高。这类大体积数据不仅容易超出大模型的单次接收上限，即便勉强传入，也会因信息过载大幅拉低AI对敏感信息的识别精度。


核心诉求是“通过简单问答即可完成敏感信息匹配、获取准确结果”，无需额外增加复杂操作。具体解决方案围绕“数据分割”展开，核心逻辑如下：


1. 核心前提：敏感信息匹配无需依赖完整响应数据包，仅需确保包含敏感信息的片段在AI可接收的有效数据大小范围内；
2. 分割逻辑：采用“基础分割 + 重叠校验”机制
   - 示例：假设某请求的响应数据量为1000KB，AI单次最多能稳定处理700KB的数据：
     - 第一次分割：覆盖0KB - 500KB的内容；
     - 第二次分割：覆盖500KB - 1000KB的内容；
     - 第三次分割：覆盖300KB - 800KB的内容(重叠校验，避免敏感信息落在分割边界被截断)。


通过这套方案，既能适配大模型的接收上限，又能保证AI对敏感信息的识别准确性，无需额外增加复杂流程。


## 0x07 流程展示
1. 获取页面网络请求：输入目标URL，如http://home.example.com/index；

3. 深度发现资源：正则资源提取器插件(如React、Vue、Angular的资源)；

5. 提取基础URL：从目标URL提取基础URL插件、从历史记录提取基础URL插件、利用正则表达式提取基础URL插件；
6. 提取基础API路径：直接指定基础API路径插件、通过关键字提取基础API路径插件、从首页网络请求(非JS资源)提取基础API路径插件、从子域名提取基础API路径插件、AI基础API路径提取器插件；
7. 提取服务路径：直接指定服务路径插件、从API路径分段组合中提取服务路径插件；
8. 提取未找到页面(404页面)：从基础API路径生成404页面插件、从基础API路径和服务路径生成404页面插件；

10. 执行自定义业务：API验证插件、AI站点定性插件、AI登录分析插件、AI公共API分析插件、自定义请求插件、正则敏感信息提取插件、AI敏感信息提取插件(概念阶段，暂不开发实现)、AI API参数获取插件(概念阶段，暂不开发实现)；
