# ApiRed Agent 优化方案 - 第 v1.0 版

## 摘要

本文档为 **ApiRed - Red Team API Security Scanner** 的第 v1.0 版优化方案，聚焦于当前项目现状与国际竞品分析，提出可落地、可版本化、可持续迭代的架构优化建议。

---

## 1. 背景

### 1.1 项目定位

ApiRed 是一款专为红队视角设计的 API 安全扫描器，核心理念：
- **黑盒发现**：不依赖目标提供文档，完全黑盒发现
- **影子 API**：从 JS/Webpack 挖掘未公开端点
- **攻击链路**：模拟真实攻击链路：发现 → 探测 → 利用

### 1.2 当前架构

```
ApiRed v3.0
├── main.py                     # 统一入口
├── core/
│   ├── scanner.py             # 主扫描器 (Stage1-5)
│   ├── engine.py              # ScanEngine 统一引擎
│   ├── pipeline.py            # 处理流水线
│   ├── agents/                # AI Agent 系统
│   ├── collectors/            # 信息采集
│   ├── analyzers/            # 分析引擎
│   ├── testers/              # 测试模块
│   ├── security/             # 安全检测
│   ├── ai/                   # AI 引擎
│   └── exporters/            # 导出模块
└── tests/                    # 测试套件 (49 tests)
```

### 1.3 当前能力

| 阶段 | 能力 | 完成度 |
|------|------|--------|
| Stage1 | JS 采集 | 70% |
| Stage2 | API 提取 | 90% |
| Stage3 | API 测试 | 85% |
| Stage4 | 漏洞验证 | 80% |
| Stage5 | 报告生成 | 90% |
| AI 增强 | LLM 端点预测 | 70% |

## 2. 当前项目问题

### 2.1 P0 阻塞问题

| 问题 | 位置 | 影响 |
|------|------|------|
| Scanner 与 Engine 双入口 | `scanner.py` vs `engine.py` | 架构冗余，维护困难 |
| Agent 实现不完整 | `analyzer_agent.py`, `tester_agent.py` | AI 增强功能形同虚设 |
| 漏洞测试用例覆盖不足 | `testers/vulnerability_tester.py` | 仅支持 SQLi/XSS/SSRF，缺 OWASP Top10 |

### 2.2 P1 严重问题

| 问题 | 位置 | 影响 |
|------|------|------|
| 并发控制不够精细 | `concurrency.py` | 高并发时可能触发目标限流 |
| 增量扫描状态不完整 | `incremental_scanner.py` | 检查点恢复可能丢失数据 |
| 报告导出格式单一 | `exporters/` | 缺少 Markdown/CSV 支持 |

### 2.3 P2 优化问题

| 问题 | 位置 | 改进方向 |
|------|------|----------|
| JS 解析仅正则 | `js_collector.py` | AST 解析（esprima）效果更好 |
| 响应聚类 O(n²) | `response_cluster.py` | 优化算法复杂度 |
| 缺乏真实目标测试 | - | 靶标环境验证 |

## 3. 外部研究结论

### 3.1 竞品分析

| 工具 | GitHub 星 | 语言 | 核心能力 | ApiRed 差距 |
|------|-----------|------|----------|-------------|
| **Noir (OWASP)** | 1.1k | Crystal | SAST-to-DAST 桥接，LLM 端点发现 | 缺乏源码分析能力 |
| **Akto** | 1.5k | Java | 1000+ 测试用例，CI/CD 集成 | 测试用例库不足 |
| **Metlo** | 1.8k | Go | 流量分析，OpenAPI 生成 | 需部署 agent，纯黑盒弱 |
| **Cherrybomb** | 1.2k | Rust | OpenAPI 验证，Brewery DSL | 仅验证，缺乏发现 |

### 3.2 关键技术趋势

1. **LLM 驱动的智能发现**
   - Noir 使用 LLM 从源码/JS 中推断 API 端点
   - 端点预测准确率 > 60%

2. **SAST-to-DAST 桥接**
   - 从源码静态分析到动态测试的自动化
   - 减少误报，提高效率

3. **测试用例库标准化**
   - Akto 的 1000+ 测试用例覆盖 OWASP API Top 10
   - 支持 CI/CD 集成

4. **增量扫描与断点续扫**
   - 支持大规模目标的分布式扫描
   - 检查点恢复机制成熟

### 3.3 可借鉴的技术方案

| 来源 | 可借鉴能力 | 适配难度 |
|------|-----------|----------|
| Noir | LLM 端点预测，SAST-to-DAST | 中 |
| Akto | 测试用例库，CI/CD 集成 | 低 |
| Metlo | OpenAPI 自动生成 | 低 |
| 微信公众号 | JS 逆向/小程序反编译思路 | 中 |

**核心能力**:

**可借鉴点**:
- 测试用例的 YAML 格式定义

### 10.1 多 Agent 协同架构

```
┌─────────────────────────────────────────────────────┐
│               AgentCoordinator                        │

│  │         Orchestrator Agent                   │   │
│  │  - 任务分解                                  │   │
│  │  - 进度协调                                  │   │
│  │  - 结果聚合                                  │   │
│  └─────────────────────────────────────────────┘   │
│                         ↓                           │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐        │
│  │ Scanner  │  │ Analyzer │  │ Tester   │        │
│  │ Agent    │  │ Agent    │  │ Agent    │        │
│  │          │  │          │  │          │        │
│  │ LLM 预测 │  │ 风险评估 │  │ 漏洞验证 │        │
│  │ 端点发现 │  │ 链路分析 │  │ 载荷生成 │        │

```

### 10.2 Agent 能力定义

| Agent | 核心能力 | LLM 调用场景 |
|-------|---------|-------------|

| AnalyzerAgent | 风险评估、链路分析 | 漏洞研判、误报过滤 |
| TesterAgent | 漏洞验证、载荷生成 | Bypass 策略生成 |
